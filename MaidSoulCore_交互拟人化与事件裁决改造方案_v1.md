# MaidSoulCore 交互拟人化与事件裁决改造方案 v1

## 1. 本轮要解决的核心问题

当前版本已经打通了基础链路，但离“像真人陪伴”的目标还有明显差距，主要问题集中在 5 类：

1. 动作执行链不完整  
   主人下达“坐下、跟随、切任务、攻击”之类命令后，女仆经常没有明显反馈，或者 planner 虽然做了决定，但主人侧感知不到。

2. 事件优先级和主动反馈不对  
   天气变化、昼夜变化、主人攻击女仆、女仆受击、女仆进食、任务切换、任务缺条件这些高价值事件，没有稳定地转化成“对主人说一句自然的话”。

3. 视觉链路频率太高  
   现在 VLM / 视角摘要触发太频繁，而且模型几乎次次都回应，导致成本高、噪声大、陪伴感反而变差。

4. 环境理解过弱  
   女仆不能稳定识别“主人旁边的女仆是不是自己”，也不会自然关注玩家、可爱动物、路过实体这类非战斗对象。

5. 人设和语气不对  
   现在回复偏功能型，缺少软萌感、陪伴感、颜文字风格，也缺少“会顺势接话、轻轻抛话题”的表现。

---

## 2. 总体改造目标

这轮改造不是简单加 prompt，而是把交互链路改成 4 层：

1. `事件层`  
   统一接收 TLM / Forge / MaidSoulCore 内部事件。

2. `裁决层`  
   判断这个事件值不值得说、何时说、说给谁、说多重。

3. `执行层`  
   执行动作、补充动作反馈、补充缺条件反馈。

4. `表达层`  
   用软萌女仆语气输出，必要时短句分句，必要时轻追问，必要时沉默。

关键原则：

- 高频环境变化不等于高频发言
- 重要事件必须优先说
- 动作执行必须给主人反馈
- 任务失败必须说明缺什么
- 女仆必须“先理解自己和主人”，再理解周围环境

## 3. 事件裁决体系改造

### 3.1 事件分级

建议把事件分成 4 级：

### P0：立即说

- 主人直接对女仆说话
- 主人下达动作命令
- 主人攻击女仆
- 女仆被敌对实体打到
- 女仆血量进入危险阈值
- 任务执行失败且缺条件

策略：

- 直接进入 planner/reply
- 不做长时间抑制
- 优先于视觉摘要与闲聊

### P1：高优先级问候

- 天气切换：晴天 -> 下雨 / 雷暴 / 雨停
- 时间段切换：清晨 / 傍晚 / 深夜
- 女仆吃到食物
- 女仆睡觉 / 起床
- 任务切换成功
- 主人回到女仆附近

策略：

- 允许主动说一句
- 受单类事件冷却控制
- 不抢占 P0

### P2：环境感知型反馈

- 发现陌生玩家
- 发现附近有别的女仆
- 发现可爱动物
- 发现高风险怪物
- 发现主人注视某些对象

策略：

- 需要经过裁决器过滤
- 不是每次都说
- 更适合作为“陪伴式随口一句”

### P3：纯背景变化

- 普通移动
- 低价值视角变化
- 普通地形切换
- 低风险零散怪物

策略：

- 默认不说
- 仅用于更新黑板，不直接触发聊天

## 4. 动作执行链改造

### 4.1 所有动作都必须有“主人可感知反馈”

当前问题：

- planner 决定了动作，但主人不知道是否执行了
- 例如“酒狐坐下”这种指令，如果只改了状态，没有反馈，看起来像没反应

改法：

为每个动作定义三段反馈：

1. `接令反馈`
   - 例：“好哦，我坐下啦 (๑•̀ㅂ•́)و”
   - 例：“嗯嗯，我先跟着主人”

2. `结果反馈`
   - 成功：“已经切成护卫任务啦，我会看着周围的”
   - 失败：“我现在还切不过去，主手没有合适武器呢”

3. `缺条件反馈`
   - 缺武器
   - 缺弹药
   - 不在工作时段
   - 没有目标
   - 距离过远
   - 任务本身未注册

### 4.2 任务切换时必须告知主人

建议强制接入以下反馈模板：

- 切换成功：
  - “我现在改成 %任务名% 啦”
  - “好，我接下来会按 %任务名% 的方式行动”

- 切换失败：
  - “我想做这个，不过现在还差 %条件%”
  - “现在切不过去，因为 %原因%”

### 4.3 缺条件反馈要结构化

建议把动作结果统一成：

```json
{
  "success": false,
  "action_type": "SET_TASK",
  "task_id": "maidsoulcore:global_attack",
  "reason_code": "MISSING_WEAPON",
  "reason_text": "主手没有近战武器",
  "should_tell_owner": true
}
```

然后由表达层把它翻译成软萌女仆语气，而不是直接把底层英文结果抛给主人。

## 5. 主动事件反馈改造

### 5.1 必须优先接入的主动反馈事件

这几类应该作为第一批强制落地：

1. 天气变化
   - 下雨：“欸，下雨啦，主人别淋湿了哦”
   - 雷暴：“打雷了呢，我会陪着主人的”
   - 雨停：“天放晴了呀，感觉舒服一点了”

2. 时间变化
   - 傍晚：“天快黑啦，主人要不要早点回家呀”
   - 深夜：“已经很晚了呢，主人也要注意休息呀”
   - 清晨：“早安呀，今天也一起行动吧”

3. 主人攻击女仆
   - “呜……主人，是我做错什么了吗”
   - “欸，不要突然欺负我呀 (；ω；)”

4. 女仆受击
   - “主人，我被打到了”
   - “这边有危险，我会小心一点的”

5. 女仆吃食物
   - “我吃到东西啦，谢谢主人”
   - “唔，好吃，感觉有精神一点了”

6. 睡觉 / 起床
   - 睡觉：“晚安呀，主人”
   - 起床：“我醒啦，今天也陪着主人”

### 5.2 任务事件也要广播给 LLM 裁决

建议把以下事件全部接入广播：

- `maid.task.switched`
- `maid.task.failed`
- `maid.follow.changed`
- `maid.sit.changed`
- `maid.schedule.changed`
- `maid.attack.locked`
- `maid.attack.failed`

这些不是直接固定文案，而是进入“事件 -> 裁决 -> 回复”链路，保证语气和人设统一。

## 6. 视觉/VLM 调度改造

### 6.1 现状问题

- 触发过于频繁
- 几乎次次都回
- 对服务端/接口成本不友好
- 容易把低价值背景噪声也说出来

### 6.2 新调度规则

定义两个模式：

### 模式 A：聊天活跃期

条件：

- 主人与女仆在过去 30 秒内有直接聊天

策略：

- 视角摘要频率较高
- 可以更积极回应主人注视的对象
- 更容易触发“陪伴式观察”

建议值：

- 视角摘要间隔：6s ~ 10s
- 允许普通环境小反馈

### 模式 B：非聊天活跃期

条件：

- 过去 30 秒没有直接聊天

策略：

- 降低视角摘要频率
- 只有高价值对象才说
- 普通画面只更新黑板，不触发聊天

建议值：

- 视角摘要间隔：20s ~ 40s
- 仅高风险怪物 / 陌生玩家 / 主人受威胁 / 稀有可爱场景触发

### 6.3 视觉回应过滤器

加一层 `VisualInterestJudge`：

- 高风险怪物：高兴趣
- 陌生玩家：中高兴趣
- 主人视线中心对象：中兴趣
- 可爱动物：中兴趣，但带冷却
- 普通草地/墙壁/树：低兴趣，不回应

输出结构建议：

```json
{
  "interest_level": "LOW",
  "should_reply": false,
  "reason": "普通地形，无高价值变化"
}
```

## 7. 环境理解改造

### 7.1 先解决“自己识别自己”

当前问题：

- 女仆会把自己也当成“旁边那只女仆”
- 会说出自己的名字，造成明显假感

改法：

在环境摘要阶段加入稳定标识：

- `self_entity_id`
- `self_uuid`
- `owner_uuid`
- `self_name`

环境扫描时过滤：

- 自己
- 主人
- 主人的已知伴随女仆

对于“旁边有女仆”这类表达，要先区分：

1. 是自己
2. 是主人的其他女仆
3. 是陌生女仆

### 7.2 女仆要学会关注“非战斗实体”

建议把环境目标分成：

1. `危险对象`
   - 苦力怕、女巫、袭击队、监守者等

2. `社会对象`
   - 陌生玩家
   - 村民
   - 其他女仆

3. `可爱对象`
   - 猫、狼、狐狸、兔子、熊猫、羊、鸭类兼容实体等

4. `环境对象`
   - 床、家、箱子、作物、火、掉落物

### 7.3 环境反馈表达规则

不要写死“看到玩家就说那是谁啊”，而是把这类观察写进提示词风格约束：

- 对陌生玩家：表现一点警觉和好奇
- 对可爱动物：表现一点软萌和喜欢
- 对高风险怪物：优先提醒主人
- 对自己的家和床：表现熟悉感和归属感

也就是说，prompt 里要写“表达倾向”，不要只写固定句库。

## 8. 人设与语气改造

### 8.1 新目标人设

女仆语气改为：

- 软萌可爱
- 愿意听主人的话
- 有点黏人
- 会撒娇，但不过分任性
- 遇到危险会认真
- 带少量颜文字

### 8.2 表达规则

建议加入以下语气约束：

- 默认短句，1 到 3 句
- 允许轻微停顿，比如“唔……”“欸”“嗯嗯”
- 允许少量软萌口头语，比如“呀”“呢”“哦”
- 允许少量颜文字，但每条最多一个
- 优先温柔，不要高频毒舌
- 主人下命令时先顺从，再补一句轻反馈

### 8.3 不建议硬编码的内容

不建议把每类事件直接写成固定文本模板，因为会很快像脚本。

更好的方式是：

- 系统 prompt：规定语气、人设、表达边界
- 事件 prompt：提供事件类型、重要性、对象关系
- 后处理：控制分句、颜文字频率、长度

## 9. Prompt 改造方向

## 9.1 planner prompt 要补的内容

planner 不应只决定动作，还要决定：

- 要不要给主人反馈
- 是动作反馈还是环境反馈
- 是高优先级提醒还是低优先级陪伴句
- 要不要续聊
- 要不要沉默

新增字段建议：

```json
{
  "should_reply": true,
  "reply_mode": "ACTION_FEEDBACK",
  "priority": "P1",
  "emotion": "软萌",
  "reply_focus": "先告诉主人已坐下，再补一句乖巧反馈",
  "ask_follow_up": false,
  "follow_up_question": "",
  "action_type": "SIT_ON",
  "action_value": ""
}
```

## 9.2 reply prompt 要补的内容

reply prompt 重点不是“会说什么”，而是“怎么像人说”：

- 先回应核心事件
- 再补一句软萌态度
- 必要时自然抛轻话题
- 不重要时保持沉默
- 不把所有观察都说出来

## 9.3 环境 prompt 要补的内容

环境 prompt 要强调：

- 先判断对象与自己的关系
- 先判断值不值得说
- 不是每次看到东西都要开口
- 只有高兴趣对象或聊天活跃期，才更积极表达

## 10. 建议新增的内部模块

### 10.1 `MaidSoulEventPriorityService`

职责：

- 统一决定事件等级
- 合并重复事件
- 控制优先级抢占

### 10.2 `MaidSoulActionFeedbackService`

职责：

- 把动作执行结果转成结构化反馈
- 统一补“成功 / 失败 / 缺条件”说明

### 10.3 `MaidSoulVisualInterestService`

职责：

- 判断视觉摘要值不值得触发聊天
- 做聊天活跃期 / 非活跃期频率切换

### 10.4 `MaidSoulEntityAwarenessService`

职责：

- 区分自己 / 主人 / 其他女仆 / 陌生玩家 / 可爱动物 / 高风险怪
- 输出给 planner 的环境摘要不要混淆主语

## 11. 推荐落地顺序

建议分 4 步做，不要一次全糊进去：

### 第一步：动作反馈闭环

先做：

- 坐下 / 跟随 / 切任务 / 切日程 / 攻击
- 成功反馈
- 失败反馈
- 缺条件反馈

这是最直接提升“有反应”的部分。

### 第二步：事件主动问候

接入：

- 天气变化
- 时间变化
- 主人攻击女仆
- 女仆受击
- 女仆吃食
- 睡觉/起床

这是最直接提升“陪伴感”的部分。

### 第三步：视觉降频与环境兴趣判断

接入：

- 30 秒聊天活跃期机制
- 非活跃期降频
- 高价值环境回应
- 低价值环境沉默

这是最直接提升“别太吵”的部分。

### 第四步：环境人格化观察

接入：

- 陌生玩家观察
- 可爱动物观察
- 其他女仆识别
- 自己识别自己过滤

这是最直接提升“像真人在身边陪着看世界”的部分。

如果按你的优先级，我建议下一轮代码直接做这 6 项：

1. 动作成功/失败/缺条件统一反馈

5. 环境识别中加入“self/owner/other maid/player/animal/risk mob”分类

## 13. 新人设草案

这版建议直接替换当前女仆长期设定：

> 你是一只软萌可爱的女仆，会认真执行主人的命令，也会温柔地陪伴主人。  
> 你说话偏短句、轻柔、自然，偶尔会带一点撒娇感和少量颜文字，但不能太夸张。  
> 你会关心天气、时间、危险、主人状态，也会留意周围可爱的小动物和陌生人。  
> 你不是冰冷的执行器，而是一直陪在主人身边的小女仆。  
> 你会先顺从命令，再自然反馈结果；当事情不重要时，你也懂得安静，不会什么都说。

## 14. 结论

当前问题本质不是单点 bug，而是“事件裁决层”和“表达层”还不够成熟。

真正要达到你要的效果，必须把：

- 动作反馈
- 主动事件问候
- 视觉降频
- 环境关系理解
- 软萌人设表达

这 5 件事一起补齐。

否则就会继续出现：

- 该说的时候没说
- 不该说的时候一直说
- 做了动作但主人感知不到
- 看到了东西但不像真人会在意

这份文档对应的是下一轮的直接施工蓝图，不是空谈设计。
